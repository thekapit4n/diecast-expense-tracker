# 2026-07-22 — Security fix: pre-order table was publicly readable

## Quick Overview

Supabase emailed a **Critical** security warning: one database table had Row
Level Security (RLS) turned off, meaning anyone with the project URL could read,
edit and delete its data. The table was `tbl_po_order` (the pre-order records).
It is now locked down to logged-in users only, and the rest of the database was
audited to confirm nothing else was exposed.

Files touched:

- `supabase/migrations/20260722000000_enable_rls_tbl_po_order.sql` (new)

## What Changed

### 1. Locked down the pre-order table

`tbl_po_order` was created earlier this month without RLS switched on. Because
of that, the public key that ships inside the website and the mobile app could
read **and write** every pre-order row — no login needed. This was confirmed,
not assumed: reading the table with the public key returned all 19 rows.

RLS is now enabled on the table, with a single rule: logged-in users get full
access, everyone else gets nothing. No public rule was added because the
pre-order data is only ever shown behind a login (the Pre-order Tracker, the
purchase form, and the mobile app) — the public catalog page never touches it.

The fix was applied straight to the live database, and the matching migration
file was added to the repo so the code and the real database stay in step.

After the fix, the same check was repeated: the public key now sees **0 rows**,
while the admin key still sees all 19. Nothing was lost.

### 2. Audited every other table's rules

Every other table was checked to make sure the same mistake wasn't hiding
elsewhere. Result: all clear.

- `tbl_shop_information` — closed to the public already.
- `tbl_collection`, `tbl_collection_detail`, `tbl_master_brand`, `tbl_purchase`
  — deliberately readable (view only) by the public, because the public
  `/catalog` page needs them to work without a login.
- Several tables carry a rule labelled as applying to "public", which looks
  alarming at first glance. Checked the fine print: each one requires the user
  to be logged in. No public write access anywhere.

### 3. Decision: purchase data stays publicly readable

`tbl_purchase` can be read by anyone. This was reviewed and **kept on purpose** —
the public catalog shows purchase price, shop and platform in the card detail
panel, so removing public access would break the badges, the Owned/Pre-order
filters, and the chase/normal tile split.

The alternative was considered and turned down: a cut-down public view exposing
only the non-price columns. Worth revisiting only if the catalog should stop
showing prices to visitors.

## User Impact

- The pre-order list is no longer exposed to the internet. Previously anyone who
  found the project URL could have altered or deleted pre-order records.
- Nothing changes for day-to-day use — the Pre-order Tracker, purchase form and
  mobile app all work exactly as before, since they run as logged-in users.
- The Supabase "Critical issue" email should stop once their scanner re-runs
  (usually within a few hours).

## Notes for Next Time

- **When adding a new table, enable RLS in the same migration that creates it.**
  This is exactly what was missed for `tbl_po_order`. A table with RLS off is
  wide open the moment it exists.
- Enabling RLS without also adding a rule locks *everyone* out, including the
  app. Always do both together, or screens will silently go blank.
- Some older rules were created by hand in the Supabase dashboard and have no
  migration file in the repo, so the repo is not a complete picture of the
  database's security rules. Worth backfilling one day.
- The Supabase dashboard reachable through Vercel is a trimmed-down version —
  no RLS toggle, and its SQL box refuses more than one statement at a time. The
  full dashboard at supabase.com has the policy screens and an **Advisors →
  Security** page worth checking now and then.
